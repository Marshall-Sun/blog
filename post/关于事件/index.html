事件定义

事件是某事发生的信号，所有的 DOM 节点都会生成这样的信号（但不只有 DOM 可以产生事件）。比如：

  • 用户在某个元素上点击鼠标 —— click
  • 用户在键盘中按下某个按键 —— keydown
  • 用户调整浏览器的大小或者关闭浏览器窗口 —— resize, close
  • 一个网页停止加载 —— DOMContentLoaded
  • 提交表单 —— submit
  • 发生错误 —— error

事件处理器

每个可用的事件都会有一个事件处理器（Event handler，事件监听器），也就是事件触发时会运行的代码块。

当我们定义了一个用来回应事件被激发的代码块时，我们就注册了一个事件处理器。

事件处理器有时也叫事件监听器，因此它包含两部分工作：监听事件是否发生，然后处理事件并做出回应。

主要有三种分配事件处理器的方法：行内特性、DOM 属性、addEventListener。

行内特性

事件处理器可以设置在 HTML 中名为 on<event> 的特性（attribute）中。

行内特性被认为是不好的做法，因为混用 HTML 和 JavaScript 在复杂的页面中会导致难以管理和效率低下。

冒泡和捕获

冒泡（Bubbling）和捕获（Capturing）都是事件传播时的阶段。

事件冒泡

当一个事件发生在一个元素上，它会首先运行在该元素上的事件处理器，然后运行其父元素上的事件处理器，然后一直向上到其他祖先上的事件处理器。

假设有 3 层嵌套 FORM > DIV > P，它们各自拥有一个事件处理器：

FORM
DIV

P

点击内部的 <p> 会首先运行 onclick：

  1. 在该 <p> 上的。
  2. 然后是外部 <div> 上的。
  3. 然后是外部 <form> 上的。
  4. 以此类推，直到最后的 document 对象。

因此，如果我们点击 <p>，那么我们将看到 3 个 alert：p → div → form。

这个过程被称为冒泡，因为事件从内部元素“冒泡”到所有父级，就像在水里的气泡一样。

event.target

在上文中，我们知道了事件对象包括了这个事件的详细信息。那么一个事件冒泡触发了一系列的事件，父元素上的事件对象可以获取子元素信息吗？

答案是可以的，引发事件的那个嵌套层级最深的子元素被称为目标元素，可以通过 event.target 访问。它主要被应用于下文的事件委托中。

停止冒泡

冒泡事件从目标元素开始向上冒泡。通常，它会一直上升到 <html>，然后再到 document 对象，有些事件甚至会到达 window，它们会调用路径上所有的事件处理器。

我们可以通过 event.stopPropagation() 决定事件已经被完全处理，并停止向上冒泡。

事件捕获

DOM 事件分发标准一共有三个阶段，事件遵守以下步骤进行传播：

  1. 捕获阶段 —— 事件（从 Window）向下走近元素。
  2. 目标阶段 —— 事件到达目标元素。
  3. 冒泡阶段 —— 事件从元素上开始冒泡。

容易得出，事件捕获与事件冒泡的方向是相反的，假设有 3 层嵌套 FORM > DIV > P：

如果你点击了 <p>，那么事件触发的顺序是：

  1. HTML → BODY → FORM → DIV（捕获阶段）
  2. P（目标阶段，触发两次，因为有两个事件处理器：捕获和冒泡）
  3. DIV → FORM → BODY → HTML（冒泡阶段）

事件委托

事件委托（Event delegation）是一种强大的事件处理模式。

如果我们有许多以类似方式处理的元素，那么就不必为每个元素分配一个处理程序 —— 而是将单个处理程序放在它们的共同祖先上。

在处理程序中，我们获取 event.target 以查看事件实际发生的位置并进行处理。

一个很好的例子是一系列列表项 <li>，如何让每个列表项被点击时弹出对应信息？

  • 低情商：将每个列表元素加上不同的事件处理器。
  • 高情商：事件委托，将事件处理器设置在父元素 <ul> 上，这样事件就会从列表项冒泡到它那里。

事件委托真的很酷！这是 DOM 事件最有用的模式之一。

我们可以总结出事件委托的三个步骤：

  1. 在共同的父元素上加上事件处理器。
  2. 在处理函数中检查源元素 event.target。
  3. 如果事件发生在对的元素内，那么处理该事件。

优缺点

优点：

  • 节省内存，无需添加许多事件处理器。
  • 更少的代码，添加/移除元素时，无需添加/移除事件处理器。
  • 支持 DOM 修改，我们可以使用 innerHTML 等方法来批量修改 DOM 元素。

缺点：

  • 事件必须冒泡。而有些事件不会冒泡。此外，子元素不应该使用 event.stopPropagation()。
  • 可能会增加 CPU 负载，但通常负载可以忽略不计。

小结

  • 事件是某事发生的信号。

  • 每个可用的事件都会有一个事件处理器，它包含两部分工作：监听事件是否发生，然后处理事件并做出回应。

  • 主要有三种分配事件处理器的方法：行内特性、DOM 属性、addEventListener。

  • 事件对象包括了这个事件的详细信息，其中的 event.target 属性指向发生该事件的嵌套最深的元素。

  • 事件遵守以下步骤进行传播：

    1. 捕获阶段 —— 事件（从 Window）向下走近元素。
    2. 目标阶段 —— 事件到达目标元素。
    3. 冒泡阶段 —— 事件从元素上开始冒泡。

  • addEventListener 默认监听冒泡阶段，要让它监听捕获阶段，可以分配 addEventListener(..., true)（true 是 { capture: true } 的一个简写形式）。

  • 可以通过调用 event.stopPropagation() 来停止事件，但不建议这样做，因为它可能会影响事件委托。

  • 事件委托是一种强大的事件处理模式，是 DOM 事件最有用的模式之一，有三个步骤：

    1. 在共同的父元素上加上事件处理器。
    2. 在处理函数中检查源元素 event.target。
    3. 如果事件发生在对的元素内，那么处理该事件。
