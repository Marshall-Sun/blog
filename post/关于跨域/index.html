同源策略

定义

所谓 源，可以指 URL。

简单来看某个 URL 组成：https://gist.github.com:443/abc，在这里：

名称 举例
协议 http、https
域名 github.com、msun.work
端口 80、443

如果 URL 未标明端口号，那么 http 默认是 80，https 默认是 443。

如果两个 URL 协议、域名、端口三者一致，那么它们同源。

目的

同源政策的目的，是为了保证用户信息的安全，防止恶意的网站窃取数据。

设想这样一种情况：A 是银行网站，用户登录以后，又去浏览其他网站。如果其他网站可以读取 A 网站的 Cookie，会发生什么？

很显然，如果 Cookie 包含隐私（比如存款总额），这些信息就会泄漏。更可怕的是，Cookie 往往用来保存用户的登录状态，如果用户没有退出登录，其他网站就可以冒充用户，为所欲为。

由此可见，同源政策是必需的，否则 Cookie 可以共享，互联网就毫无安全可言了。

跨域

定义

随着互联网的发展，同源政策越来越严格。如果非同源，共有三种行为受到限制：

  • DOM 无法获得。
  • AJAX 请求不能发送。
  • Cookie、Web Storage 和 IndexDB 无法读取。

这就是跨域问题，虽然这些限制是必要的，但是有时很不方便，合理的用途也受到影响。

常见表现

  • 前后端服务器分开部署，如端口不一致。
  • 本地开发（本地预览项目时，请求了测试服务器的数据）。
  • 调取了第三方平台的接口。

跨域常用解决方案

JSONP

JSONP（JSON with Padding）是服务器与客户端跨域通信的常用方法。

它的基本原理是 <script> 可以具有任何域的 src 属性，例如 <script src="http://another.com/…">。因此通过添加一个 <script> 元素，向服务器请求 JSON 数据，服务器收到请求后，将数据放在回调函数里传回。

实现方法

假设在我们的网站，需要以这种方式从 http://another.com 网站获取天气数据：

  1. 先声明一个全局函数来接收数据，例如 gotWeather。

  2. 然后创建一个 <script> 标签，设置它的 src="http://another.com/weather.json?callback=gotWeather"，其中参数 callback 为上一步的函数名称 gotWeather。

  3. 远程服务器 another.com 调用 gotWeather，发送它想让我们接收的数据。

  4. 服务器执行 gotWeather 时，因为它是客户端的函数，我们就有了需要的数据。

在 Angular 中，可以引入 HttpClientJsonpModule 来直接使用 JSONP，见 官方文档。

特点

  • 兼容性好，能解决主流浏览器跨域问题。
  • 仅支持 GET 请求。
  • 不安全，可能会遭受 XSS 攻击。具体可见 XSS。

CORS

跨域资源共享（CORS，Cross-origin resource sharing）是一种机制，它允许浏览器向跨域服务器发出 XMLHttpRequest 或 Fetch 请求。

CORS 需要浏览器和服务器同时支持。支持后整个 CORS 通信过程，都是浏览器自动完成，不需要用户参与。浏览器一旦发现 AJAX 请求跨源，就会自动添加一些附加的头信息，随后和同源通信没有差别。

因为目前各大浏览器都支持了 CORS，所以只要服务器支持 CORS 就可以跨域通信。

实现方法

如果一个 HTTP 请求是跨域的，浏览器始终会向 HTTP header 中添加 Origin 字段。

例如，如果我们从 https://javascript.info/page 请求 https://anywhere.com/request，请求报文的 HTTP header 如下：

GET /request
Host: anywhere.com
Origin: https://javascript.info
...

服务器可以检查 Origin，如果同意接受这样的请求，就会在 HTTP 响应报文中的 header 添加一个属性 Access-Control-Allow-Origin。它的值为允许的源（在我们的示例中是 https://javascript.info），或者一个星号 *，表示所有网站都可以访问资源。

这是一个带有服务器许可的响应报文 HTTP header：

200 OK
Content-Type:text/html; charset=UTF-8
Access-Control-Allow-Origin: https://javascript.info

浏览器接受到 HTTP 响应报文，检查 Access-Control-Allow-Origin 的值是否与 Origin 的值匹配，如果存在，则允许 JavaScript 访问响应，否则将失败并报错。

分类

浏览器将 CORS 请求分成两类：简单请求、非简单请求。只要同时满足以下两大条件，就属于简单请求。

  1. HTTP 方法是下列之一：
  • HEAD
  • GET
  • POST
  1. HTTP header 只具有：
  • Accept
  • Accept-Language
  • Content-Language
  • Last-Event-ID

凡是不同时满足上面两个条件，就属于非简单请求。一个非简单请求不仅包含通讯内容的请求，同时也包含预请求。

简单请求和非简单请求的区别：

  • 简单请求的发送和普通的请求区别不大，就是上文描述的方法。
  • 非简单请求不止发送一个请求。其中最先发送的是一种“预请求”，而服务端也需要返回“预回应”作为相应。预请求实际上是对服务端的一种权限请求，只有当预请求成功返回，才发出实际请求，应用先前的简单请求步骤。

WebSocket

WebSocket 是 HTML5 新增的一种应用层协议，它是一种客户端与服务器进行全双工通讯的网络技术。

因为不是 HTTP，所以不会有跨域问题。具体可见 关于 WebSocket。
