说下 JavaScript 的数据类型种类

八种，String, Number, Boolean, Object, Undefined, Null, Symbol, BigInt。

栈储存着原始数据类型，共七种：String, Number, Boolean, Undefined, Null, Symbol, BigInt。存储占据空间小、大小固定，属于被频繁使用数据，所以放入栈中存储。

堆储存着引用数据类型，共一种：Object。存储占据空间大、大小不固定。如果存储在栈中，将会影响程序运行的性能；引用数据类型在栈中存储了指针，该指针指向堆中该实体的起始地址。当解释器寻找引用值时，会首先检索其在栈中的地址，取得地址后从堆中获得实体。

Symbol 代表创建后独一无二且不可变的数据类型，它主要是为了解决可能的全局变量冲突。

BigInt 是一种数字类型数据，它可以表示任意精度的整数，故可以安全地存储和操作超出 Number 范围 [-(2^53 - 1), 2^53 - 1] 的整数。

Undefined 表示一个变量自然的、最原始的状态值。(值不存在)

Null 则表示一个变量被人为的设置为空对象，而不是原始状态。(值存在但是为空，没有意义)

讲下原型与原型链

  • 对象有两种：所有 Function 的实例都是函数对象，而其他的都是普通对象。
  • prototype 和 __proto__ 都是 JavaScript 在定义一个函数或对象时自动创建的预定义属性，prototype 是一个普通对象 {...}，而实例的 __proto__ 指向构造函数的 prototype。
  • 普通对象不具有 prototype 这个属性，它只有 __proto__，这是原型链查找的起点。

举一个例子：

  • 若 A 通过 new 创建了 B，则 B.__proto__ = A.prototype。
  • 执行 B.a，若在 B 中找不到 a，则会在 B.__proto__ 中，也就是 A.prototype 中查找，若 A.prototype 中仍然没有，则会继续向上查找，最终一定会找到 Object.prototype，倘若还找不到，因为 Object.prototype.__proto__ 指向 null，因此会返回 undefined。
  • 为什么万物皆空，因为原型链的顶端一定有 Object.prototype.__proto__ === null。

讲下 ECMAScript

Ecma 国际（Ecma International）是一家制定并维护计算机各种标准的国际组织。前名为欧洲计算机制造商协会（European Computer Manufacturers Association，ECMA）。

TC39 是 Ecma 国际的一个部门，专门负责制定 ECMA 第 262 号标准，标准代号为 ECMA-262，这个标准就是 ECMAScript。

JavaScript 则是一种通用脚本编程语言，它遵循了 ECMAScript 标准。换句话说，JavaScript 是 ECMAScript 的一种方言。

说下 var、let 与 const

ES2015（ES6）之前，声明变量的方法只有 var。let 与 const 的加入引入了块级作用域的概念。

它们的区别：

  • let 和 const 声明形成块作用域。
  • 同一作用域下 let 和 const 不能声明同名变量，而 var 可以。
  • var 声明的变量会挂载在 window 上，而 let 和 const 声明的变量不会。
  • var 声明变量存在变量提升，let 和 const 不存在变量提升，如果在声明之前使用这些变量就会报错，这就是暂时性死区（temporal dead zone）。
  • const 一旦声明必须赋值，不能使用 null 占位。原始数据类型声明后不能再修改，引用数据类型可以修改其属性。

说下网络安全

XSS

跨站脚本攻击（Cross-Site Scripting，为了和 CSS 区分，简称 XSS），是一种代码注入攻击。攻击者通过在目标网站上注入编写好的代码（恶意脚本），使之在用户的浏览器上运行。利用这些恶意脚本，攻击者可获取用户的敏感信息如 Cookie、SessionID 等，进而危害数据安全。

XSS 的本质是：恶意代码未经过滤，与网站正常的代码混在一起；浏览器无法分辨哪些脚本是可信的，导致恶意脚本被执行。

常用防范方法

  • HttpOnly

    JavaScript Document.cookie API 无法访问带有 HttpOnly 属性的 Cookie。

    Set-Cookie: id=a3fWa; Expires=Wed, 21 Oct 2015 07:28:00 GMT; Secure; HttpOnly

  • 输入过滤 + 转义

    一般是用于对于输入格式的检查，例如邮箱、电话号码、用户名、密码等都要按照规定的格式输入，同时应该限定一个合理的长度。

    这不仅仅是前端负责，后端也要做相同的过滤检查。因为攻击者完全可以绕过正常的输入流程，直接利用相关接口向服务器发送设置。

CSRF

跨站请求伪造（Cross-site request forgery，CSRF），是一种劫持受信任用户向服务器发送非预期请求的攻击方式。

通常情况下，CSRF 攻击是攻击者借助受害者的 Cookie 骗取服务器的信任，可以在受害者毫不知情的情况下以受害者名义伪造请求发送给受攻击服务器，从而在并未授权的情况下执行在权限保护之下的操作。

要完成一次 CSRF 攻击，受害者必须依次完成两个步骤：

  1. 登录受信任网站 A，并在本地生成 Cookie。
  2. 在不登出 A 的情况下，访问危险网站 B。

防御

  • 验证码。强制用户必须与应用进行交互，才能完成最终请求。此种方式能很好的防范，但是提高了用户操作难度。
  • 同源检测。既然 CSRF 大多来自第三方网站，那么就直接禁止外域（或者不受信任的域名）发起的请求。
  • Token。这是公认最合适的方案，因为 CSRF 攻击依赖于 Cookie。
