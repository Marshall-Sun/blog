研究Array.prototype.sort()
发表于 2020-09-10| 更新于 2020-09-10
字数总计:2k|阅读时长: 6 分钟

Array 的原型上有一个非常容易误用的方法 sort() ，它总是能让诸多 JavaScript 新手感到相当地困惑，本篇文章会先介绍它的正确使用姿势，而后讨论各家厂商是如何来实现排序。最后再来探讨 sort 常常成为 JavaScript 被抨击的原因。

前言

JavaScript 的函数用法，总是跟新手想的不一样。这也是撰写这篇文章的主要原因之一，常见的 sort 方法太过容易被误用，导致不断有人制作成 meme 图来攻击 JavaScript 很 weird。

另外这篇文章并非教学文，虽然还是会介绍语法，但主要想讨论的是「正确的语法」、「实际上的排序操作」、「为什么它让人讨厌」等等。

sort() 的用法

arr.sort([compareFunction])

可以发现 Array 上的 sort 方法是可以接受一个回调函数，MDN 把它称为 compareFunction，我们来看看实际上这个回调函数是怎么样去设计的。

我们先不管排序标准，当我们要看懂一个函数的时候，有一件最重要的事，就是看懂传入的参数代表什么。于是我们来做一个实验来看看参数到底是什么：

我们发现输出的结果完全相反，这十分不易于理解。但是本来 JavaScript 就没有规范 compareFunction 传入的参数到底该是什么顺序（这跟等等提到的算法有关）， compareFunction 只是在制定一个两数之间的比较规则。

什么是明确规则？

如果今天要比较的两个数字分别是 113、114，我们要告诉 JavaScript 的引擎，到底排序的方式是什么样的，这个时候只有三种状况：a 排在 b 前面、a 排在 b 后面、ab 位置不动。

if (a < b) {
  return -1;
}
//当a小于b的时候，把a排在b前面

if (a > b) {
    return 1;
}
//当a大于b的时候，把a排在b的后面

return 0;
//a等于b，return 0 代表不改变彼此之间的顺序

在这个明确规则的定义后，不管 a 跟 b 两数传进来的顺序为何，我们只在乎它们彼此的排序关系。这也解释了为什么同样的 compareFunction 下，尽管 Chrome 与 Firefox 接受的参数不一样，还是能排序出一样的结果。

再次注意！Compare Function 的这个回调只在乎排序两数之间的规则。

你可能会觉得有点懵，明明传进来的顺序就不一样，为什么可以这样做，这有点像是在绕口令：

我比你高，换句话说你比我矮，只要规定高矮之间谁在前面、谁在后面就好。

那当我们什么都不写的时候呢？

根据规范，预设的排序顺序是根据字串的 Unicode 编码位置（code points）而定，这也是为什么许多人直接输入arr.sort()仍然会排序（但通常排出来的跟想象中不一样）的原因。

聊聊各家实现 sort() 的算法

仔细看看 sort() 的介绍

sort()方法会原地（in place）对一个数组的所有元素进行排序，并回传此数组。排序不一定是稳定的（stable）。由于依赖执行环境的实作，所以并不能保证排序的时间及空间复杂度。

先来讲讲原地（in place）

维基百科上针对原地算法有详细的介绍，所以就直接告诉你，原地的算法会直接在原本的数组上操作元素，这样不用额外的内存空间，也就是空间复杂度为 O(1)，副作用就是会覆盖原本的数组。

再来讲讲什么是稳定（stable）

稳定的排序法在拥有相同值的状况下，排序后的位置保证会跟排序前相同：

排序前：[1, 3, 2, 5, 6, 3*]、排序后：[1, 2, 3, 3*, 5, 6]

然而不稳定的排序法可能会造成：

排序前：[1, 3, 2, 5, 6, 3*] 、排序后：[1, 2, 3*, 3, 5, 6]

你可能会觉得这有什么不妥？稳定、不稳定会造成什么问题？
我们观看以下例子就可以发现还是有些差别的。

如果依照身高（height）做排序，稳定的排序可以确保 Jonathan 永远排序在 Dio 前面，但是不稳定的排序可能最终的回传结果会是 Dio 在 Jonathan 前面。

sort() 的实现方式并没有统一规范

由于依赖执行环境的实现方式不同，所以并不能保证排序的时间及空间复杂度。

对，这也代表一件事，不同的浏览器之间的实现方式并没有统一，而且尽管是同一种浏览器，也会因为版本的不同，而有不同的排序实现方式。

可能采用不同的算法，各个算法可能是 stable 也可能是 unstable。

What is the stability of the Array.sort() method in different browsers? - Stack Overflow

  • IE6+: stable
  • Firefox < 3: unstable
  • Firefox >= 3: stable
  • Chrome < 70: unstable
  • Chrome >= 70: stable
  • Opera < 10: unstable
  • Opera >= 10: stable
  • Safari 4: stable
  • Edge: unstable for long arrays

这个答案的整理很棒，这意味着即使你传入同样的 compareFunction，最终结果在不同的浏览器，甚至是同个浏览器的不同版本都会有歧义。

由 Mathias Bynens（V8的开发者）提供的答案：

As of V8 v7.0 and Chrome 70, our Array.prototype.sortimplementation is now stable. 🎉

Previously, V8 used an unstable QuickSort for arrays with more than 10 elements. Now, V8 uses the stable TimSort algorithm.

可以发现，在旧版本的Chrome中，他们使用快速排序法（quick sort）来实作排序，但在新的 Chrome 版本里头是使用 TimSort 来实现，两者的时间复杂度都是 O(nlogn)，但是后者是稳定的排序法。

那我们刨根究底来看看 Firefox 的 stable sorting 是用什么算法呢？
答案就是合并排序法（merge sort），时间复杂度也是 O(nlogn)，这个讨论串还可以看到当时的讨论。

就像上头说的，ECMA 并没有明确规定Array.prototype.sort()的方式到底需不需要 stable，所以掀起一阵讨论。

为什么有人会使用 sort() 来抨击JavaScript

  1. 没有搞清楚 sort() 是高阶函数

    太多刚接触JavaScript 的同学，可能默认了 sort() 的行为，对于函数不够熟练。

  2. 没有阅读文档

    因为太多语言的sort()开箱即用，不会有这么一个古怪的规则「预设的排序顺序是根据字串的 Unicode编码位置（code points）而定」。

  3. 搞不懂 compare function

    如果直接的把两参数打印出来只会更加的无解，需要知道的是， sort() 方法本来就与forEach()、map()、filter() 有一个根本上的差异，后者都是透过不停地迭代往前，但sort() 并不是，试想难道一轮迭代就能结束排序了吗？

  4. 被各家实现方法不一的 stable 、 unstable 搞得非常困惑

    这没什么好说的，就真的 weird，如果要 stable 的排序，还是使用外部的函数库会比较保险。
