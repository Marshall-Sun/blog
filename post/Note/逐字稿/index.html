说下懒加载

懒加载是一种前端性能优化的方法，原理是先将图片的 src 属性设置为统一的占位图，当用户将要下拉到图片位置时再将 src 属性替换为真实的 URL，开始加载。这样就可以减小浏览器的负担。

实现方法

  1. 先将图片的真实 URL 赋给图片的某个无关属性，如 data-src，图片的 src 属性则设为统一的占位图

  2. HTMLElement.offsetTop 可以获取 HTML 元素相对于其定位父元素的顶部距离，对其进行监听，当它的值小于布局视口高度+文档滚动长度+自定义提前量时，将 data-src 属性赋给 src 属性即可

我的博客是直接使用了 lazysizes 这个库，它可以帮助实现第二步，要做的就是设置 data-src 属性，并在图片上设置 class="lazyload" 即可。

说下响应式布局

响应式布局指的是同一页面在不同屏幕尺寸下有不同的布局，这样就可以较方便地适配移动端。

响应式的实现可以有几方面：

  • 媒体查询

    设定一系列媒体查询节点，根据不同设备宽度分别应用不同的 CSS 规则来实现响应式。当重置浏览器窗口大小的过程中，页面就会根据浏览器的宽度重新渲染。

  • 百分比

    在设置元素样式的时候，除了使用 px 等绝对长度单位，也可以使用 % 这种相对长度单位。再配合媒体查询，就可以更方便地实现响应式。

    需要注意的是，子元素的 width/padding/margin（包括 padding/margin-top/buttom）如果设置百分比，都是相对于父元素的宽度；而子元素的 height 是相对于父元素的高度。

    这相对父元素的属性并不是唯一的。所以可能使布局问题变得复杂。

  • rem

    rem 也是一个相对长度单位，它是相对于根元素字体宽度变化的，这相当于提供了一个基准。因此如果修改根元素字体宽度，就可以一次性统一调整所有使用 rem 单位的 CSS 长度。

  • 视口

    vw/vh，是视觉视口宽度 window.innerWidth/innerHeight 的百分比。

通过使用媒体查询和选用百分比/rem/视口这三种相对长度单位来度量长度，可以较方便地实现响应式布局。

说下事件循环

事件循环 (Event Loop)，是运行在浏览器 / Node 环境中的一种消息通信机制，是 JavaScript 处理同步和异步操作的方法。

同步任务由执行栈处理，而异步任务由任务队列处理。

根据任务性质的不同，分为宏任务（macrotask）和微任务（microtask）。它们会被放入不同的任务队列中，且微任务要优于宏任务执行。即在执行完当前宏任务后，清空微任务队列，再执行下一个宏任务。

常见的宏任务和微任务

宏任务（macrotask）举例：

  • I/O
  • setTimeout
  • setInterval

微任务（microtask）举例：

  • Promise.then/catch/finally

运行机制

初始状态：执行栈空，微任务队列空，宏任务队列里有且只有一个 script 脚本（整体代码）。

  1. 全局上下文被推入执行栈，之后所有同步任务都在执行栈执行。

    同时异步任务被放到异步处理模块，当有了运行结果后就移入任务队列。

  2. 执行栈被清空后，处理任务队列中的异步任务。

  3. 执行渲染操作，更新界面。

上述过程循环往复，直到所有队列都清空，这就是事件循环。

说下 async 函数

JavaScript 里的异步方案的演进，是这样的：回调函数 -> Promise -> 生成器函数 -> async 函数

ES2017 标准引入了 async 函数，使得异步操作变得更加方便。

async 函数就是生成器函数的语法糖，它可以看作多个异步操作包装成的一个 Promise 对象；而 await 命令就是内部 then() 命令的语法糖。

当函数执行的时候，一旦遇到 await 就会等到微任务 then() 完成，再接着执行函数体内后面的语句。

这样，我们就可以通过同步语义编写异步操作，简化了代码。

说下跨域

同源策略

判断方法：如果两个 URL 的协议、域名、端口三者一致，那么它们同源。

同源策略是为了保证用户信息的安全，防止恶意网站窃取数据。如果缺少了同源策略，则浏览器的正常功能可能都会受到影响。

跨域问题

如果两个 URL 不同源，那么：

  1. Cookie、LocalStorage 无法读取。
  2. DOM 无法获得。
  3. 异步网络请求不能发送。

这就是跨域问题，虽然这些限制是必要的，但是有时很不方便，合理的用途也受到影响。

在开发过程中可能的跨域表现：

  • 前后端服务器分开部署，如端口不一致。
  • 本地开发（本地预览项目时，请求了测试服务器的数据）。
  • 调取了第三方平台的接口。

解决方案

JSONP

它的基本原理是 <script> 可以具有任何域的 src 属性，例如 <script src="http://another.com/…">。因此通过添加一个 <script> 元素，向服务器请求 JSON 数据，服务器收到请求后，将数据放在回调函数里传回。这样就有了需要的数据。

在 Angular 中，可以引入 HttpClientJsonpModule 来直接使用 JSONP，见 官方文档。

特点：

  • 兼容性好，能解决主流浏览器跨域问题。
  • 仅支持 GET 请求。
  • 不安全，可能会遭受 XSS 攻击，比如在 <script> 标签中注入恶意内容。

CORS

跨域资源共享（CORS，Cross-origin resource sharing）需要浏览器和服务器同时支持。支持后整个 CORS 通信过程，都是浏览器自动完成，不需要用户参与。浏览器一旦发现 AJAX 请求跨源，就会自动添加一些附加的头信息，随后和同源通信没有差别。

因为目前各大浏览器都支持了 CORS，所以只要服务器支持 CORS 就可以跨域通信。

实现方法：

如果一个 HTTP 请求是跨域的，浏览器始终会向 HTTP header 中添加 Origin 字段。

例如，如果我们从 https://javascript.info/page 请求 https://anywhere.com/request，请求报文的 HTTP header 如下：

GET /request HTTP/1.1
Host: anywhere.com
Origin: https://javascript.info
...

服务器可以检查 Origin，如果同意接受这样的请求，就会在 HTTP 响应报文中的 header 添加一个属性 Access-Control-Allow-Origin。它的值为允许的源（在我们的示例中是 https://javascript.info），或者一个星号 *，表示所有网站都可以访问资源。

这是一个带有服务器许可的响应报文 HTTP header：

200 OK
Content-Type:text/html; charset=UTF-8
Access-Control-Allow-Origin: https://javascript.info

浏览器接受到 HTTP 响应报文，检查 Access-Control-Allow-Origin 的值是否与 Origin 的值匹配，如果存在，则允许 JavaScript 访问响应，否则将失败并报错。

分类：

浏览器将 CORS 请求分成两类：简单请求、非简单请求。只要同时满足以下两大条件，就属于简单请求。否则属于非简单请求。

  1. HTTP 方法是三者之一：HEAD / GET / POST

  2. HTTP header 只具有规定的部分字段

简单请求和非简单请求的区别：

  • 简单请求的发送和普通的请求区别不大，就是上文描述的方法。
  • 非简单请求不止发送一个请求。其中最先发送的是一种“预请求”，而服务端也需要返回“预回应”作为相应。预请求实际上是对服务端的一种权限请求，只有当预请求成功返回，才发出实际请求，应用先前的简单请求步骤。

WebSocket

WebSocket 是 HTML5 新增的一种应用层协议，它是一种客户端与服务器进行全双工通讯的网络技术。

因为不是 HTTP，所以不会有跨域问题。

代理

同源策略是浏览器的标准，而如果是服务器向服务器请求就无需遵循同源策略。而代理就是通过中间服务器，让客户端与目标服务器进行非直接的连接，这样就可以实现跨域。

现在大部分框架都提供了代理的能力，底层大部分基于 http-proxy-middleware，这个库利用正则匹配寻找 HTTP 请求，如果符合要求，则在 Node.js 中自动转发到目标服务器，拿到响应数据后又转发给回客户端，实现跨域。

Angular 中集成了这一功能，使得使用更加简单，只要配置 --proxy-config 选项就可以了。具体可见 官方文档。

说下防抖和节流

防抖（debounce）：防止抖动，以免把一次事件误认为多次。N 秒内函数不会执行，若这段时间内高频事件再次被触发，则重新计时。

代码实现

节流（throttle）：控制水的流量，即事件发生的频率。指定一个 N 秒的间隔时间，不管高频事件触发多少次，只会在间隔点上执行。

有两种实现方法，区别如下：

方法 开始触发时 停止触发后
定时器实现 N 秒后执行 继续执行一次事件
时间戳实现 立刻执行 不再执行事件

使用场景

防抖：多次事件合为一次触发

  • 防止表单多次提交。
  • 输入框连续输入，减少搜索发起次数。
  • 文本编辑器实时保存，无操作一段时间后自动保存。
  • 窗口缩放，等待缩放完毕后，才重新计算 DOM 尺寸。

节流：多次事件按时间平均触发

  • 鼠标拖拽事件。
  • 鼠标滚动事件，如页面无限加载时下拉触发加载。

关于时间复杂度

算法 时间复杂度
二分查找 O(log n)
二叉树遍历 O(n)
排序二维矩阵 O(n)
归并排序 O(n log n)

二叉树遍历中的前序、中序、后序：时间复杂度是多少？

  • 时间复杂度是 O(n)，无论是前序、中序或者后序每一个节点都会访问一次，并且仅访问一次；
  • 所以就是二叉树的节点总数，也就是 O(n)的线性时间复杂度；

搜索算法：DFS、BFS 时间复杂度是多少？

  • DFS 是深度优先，BFS 是广度优先算法。
  • 不管是深度优先还是广度优先，因为访问的节点只访问一次，所以时间复杂度也是 O(n)的。（n 指的是搜索空间里面的节点总数）

说下浏览器存储与鉴权

Cookie

HTTP Cookie（Web Cookie，浏览器 Cookie）是服务器发送到用户浏览器并保存在本地的一小块数据，内部以键值对的方式来存储。它会在浏览器下次向同一服务器再发起请求时被携带并发送到服务器上。

HTTP 是无状态的，即它不会记住用户的操作，这使得保留用户状态等场景被限制，所以创造了 Cookie 弥补 HTTP 在状态管理上的不足。

使用场景

  • 会话状态管理（如保存用户登录状态、购物车、游戏分数或其它需要记录的信息）
  • 个性化设置（如用户自定义设置、主题等）
  • 浏览器行为跟踪（如跟踪分析用户行为等）

缺陷

  • 容量缺陷。存储上限为 4 KB，只能存储少量信息。
  • 性能缺陷。每次 HTTP 请求都会携带上完整的 Cookie，随着请求数增多，造成性能浪费。
  • 安全缺陷。以纯文本的形式明文在浏览器和服务器中传输，容易被非法截获和篡改。

Web Storage

Web Storage 包含 localStorage 和 sessionStorage，使浏览器能以一种比使用 Cookie 更直观的方式存储键值对。

种类 最大容量 通信方法
Cookie 4 KB 每次通信都会携带在 HTTP 头中，若保存过多数据会带来性能问题
Web Storage 5 MB 仅在客户端中保存，不参与通信
种类 生命周期
Cookie 一般由服务器生成，可设置失效时间。默认关闭浏览器后失效。
localStorage 除非被清除，否则永久保存
sessionStorage 仅在当前会话下有效，会话结束后被清除

Session

Session 字面意思上可以理解为会话，是一种记录客户端状态的机制。

与 Cookie 的区别

  • Cookie 保存在客户端，Session 保存在服务器。
  • Session 避免在客户端中存储敏感数据，相对更安全。
  • Session 一般与 Cookie 结合使用。

使用方法举例

  1. 客户端向服务器发送用户名和密码，进行登录。
  2. 服务器验证通过后，在当前会话（Session）里面保存相关数据，比如用户名，登录时间等。
  3. 服务器向客户端返回登录成功信息，HTTP 头部的 Cookie 包括了 session_id。
  4. 客户端随后的每次网络请求，HTTP 头部的 Cookie 都会包括 session_id。
  5. 服务端接受网络请求，读取 session_id，找到保存的对应 Session，由此得知用户身份。

缺陷

  • Session 保存在服务器内存中，随着认证用户的增加，服务器的消耗就会增大。
  • 因为与 Cookie 结合使用，所以继承了 Cookie 的安全缺陷，易受到 CSRF 攻击。

Token

Token（令牌）是用户身份的验证方式。当用户首次登录后，服务器生成一个 Token 并将此 Token 返回给客户端，以后客户端只需带上这个 Token 请求数据即可，无需再次提交用户名和密码。

可以把 Token 想象成通行证。用户在安全的前台验证身份（提交用户名和密码），如果成功验证了就可以取得通行证。当用户进入管制区域时（试图调用 API 获取资源），会被要求检查通行证，而不是回前台重新验证身份。

验证流程

  1. 客户端首次使用用户名与密码登录。
  2. 服务端收到登录请求，验证用户名与密码。
  3. 验证成功后，服务端会签发一个 Token，再把这个 Token 发送给客户端。
  4. 客户端收到 Token 以后把它存入 Cookie 或 localStorage。
  5. 客户端随后每次请求资源时，需要带着这个 Token。
  6. 服务端收到请求，校验 Token 的合法性，合法就放行，返回请求的数据；不合法就返回状态码 401（鉴权失败）。

优点

  • 因为不使用 Cookie，所以可以避开跨域问题。
  • 因为不使用 Cookie，所以可以避免 CSRF 攻击。
  • 因为用户信息不存储在服务器，所以是无状态的，可以在多个服务间共享。
  • 相对于 Session，不需要在服务器保留认证信息，减少服务器的消耗，提高了扩展性。

过期

出于安全考虑，Token 肯定是要有过期时间的，不然权限认证就意义不大了。Token 过期时间一般不会超过一个小时，但如果每小时重复登录以获取新的 Token，用户体验就会极差。

于是就产生了 Refresh Token，通过它来得到新的 Token。

Refresh Token 也是加密字符串，和 Token 相关联。但与用来获取资源的 Token 不同，Refresh Token 的作用仅仅是获取新的 Token。因此其作用和安全性要求都较低，所以其过期时间相对较长，可以以天为最小单位。当然如果 Refresh Token 也过期就要重新登录了。

JWT

JSON Web Token（JWT）是基于 Token 与 JSON 的认证授权机制，是目前最流行的跨域认证解决方案。它是一个很长的字符串，中间用 . 分隔成三个部分：

Header：一个 JSON 对象，描述 JWT 的元数据，使用 Base64 算法转成字符串。

Payload：一个 JSON 对象，用来存放实际需要传递的数据，使用 Base64 算法转成字符串。

注意：JWT 默认使用 Base64 算法，任何人都可以转换后读取，所以如果保存秘密信息，应该另外加密。

Signature：对前两部分的签名，防止数据篡改。

算出签名以后，把 Header、Payload、Signature 三个部分用 . 拼成一个字符串，就得到了 JWT。

和 Token 的区别

服务端验证 Token 时，需要查询数据库获取用户信息，然后验证 Token 是否有效。

而 JWT 是将 Token 与其他信息一起加密，服务端只需要使用密钥解密进行校验（校验也是 JWT 自己实现的），不需要查询数据库，因为 JWT 已经包括了用户信息和加密的数据。

使用方式

因为 JWT 是基于 Token 的，所以两者使用方式大致相同。

客户端收到服务器返回的 JWT，可以储存在 Cookie 里面，也可以储存在 localStorage。

此后，客户端每次与服务器通信，都要带上这个 JWT。可以把它放在 Cookie 里面自动发送，但是这样不能跨域，所以更好的做法是写入 HTTP 请求头部中的 Authorization 字段。

Authorization: Bearer <token>

通过调用 Angular 框架的 HttpInterceptor 接口，可以写入 HTTP 请求头部中的 Authorization 字段。这里可以参考 官方文档。

说下 Git 常用命令

  • git add：将工作区修改的内容提交到暂存区。
  • git commit：将暂存区的内容提交到本地仓库。
  • git branch：查看编辑 Git 的分支。
  • git push：上传本地仓库分支到远程仓库分支，实现同步。
  • git pull：从远程获取代码并合并本地的版本，实现同步。
  • git merge：把两个分支合并起来，产生一个额外的合并提交。
  • git rebase：将两个分支融合成一个线性的分支，产生线性的项目历史记录。
  • git reset：把当前分支指向另一个位置。
  • git revert：用一个新提交来消除一个历史提交所做的所有修改。
