说下 MVC 和 MVVM

MVC（Model-View-Controller）模式的意思是，软件可以分成三个部分：

  • 视图（View）：用户界面
  • 控制器（Controller）：业务逻辑
  • 模型（Model）：数据保存

之间所有通信都是单向的：

  1. View 传送指令到 Controller
  2. Controller 完成业务逻辑后，要求 Model 改变状态
  3. Model 将新的数据发送到 View，用户得到反馈

MVVM（Model-View-ViewModel）模式将 Controller 改名为 ViewModel，同时改变了通信方向。

  1. 采用双向绑定，将 View 与 ViewModel 联系起来，一部分出现变动，自动反映在另一部分中
  2. View 与 Model 不发生联系，都通过 ViewModel 传递
  3. View 非常薄，不部署任何业务逻辑，而 ViewModel 非常厚，所有逻辑都部署在那里

对比一下 Angular 和 React 的区别

严格来说 Angular 和 React 不是属于一类的。

因为 Angular 是一个成熟、功能完备的框架，特点是全；而 React 只是一个 UI 组件库，特点是轻。

Angular

Angular 主要由 Google 维护，它提供了非常多的开箱即用的特性：

  • 依赖注入
  • 双向绑定
  • 由 @angular/router 提供路由功能
  • 利用 @angular/common/http 实现 HTTP 请求
  • 利用 @angular/forms 创建表单

当不想花时间去挑选其他库的时候，有这么多开箱即用的特性是很方便的。

React

React 是由 Facebook 开发并维护的，它主要的特点是基于 JavaScript 的 JSX 模板，但其他的特性不多。

这意味着必须引用第三方库来实现高级功能：

  • 由 React-router 提供路由功能
  • 利用 axios / Fetch 实现 HTTP 请求
  • 利用 Redux 管理组件间的状态变化

这意味着更高的自由度，但是对于大型应用来说，不能做到开箱即用。

总之，React 和 Angular 各有各的特点，写 React 更类似去写更纯粹的 JavaScript，写 Angular 更类似去写 Web Component。

当时技术选型的时候，前端框架是怎么选择 Angular 的？

一项特定的技术是否适合具体项目使用，取决于多种因素。

它的成熟度如何，是否有稳定支持？

Angular 主要由 Google 维护；而 React 是由 Facebook 开发并维护。

在这方面两者都做的非常好。

它有什么样的特性？

Angular 主要由 Google 维护，它提供了非常多的开箱即用的特性：

  • 依赖注入
  • 双向绑定
  • 由 @angular/router 提供路由功能
  • 利用 @angular/common/http 实现 HTTP 请求
  • 利用 @angular/forms 创建表单

React 主要的特点是基于 JavaScript 的 JSX 模板，数据单向流动，但其他的特性不多。

它的生态圈如何？

由于 Angular 开箱即用的特点，社区中 Angular 相关插件相对没那么多，但是由于 Google 的大力支持，社区中相关生态还是比较可观的。

由于 React 自带的特性不多，必须引用第三方库来实现高级功能：

  • 由 React-router 提供路由功能
  • 利用 axios / Fetch 实现 HTTP 请求
  • 利用 Redux 管理组件间的状态变化

这也使得 React 及其相关库在开源社区中极为热门，资源极为丰富。

开发者上手难度

当时我们实验室大部分人都对 Angular 更熟悉，而选择 React 或者 Vue 的话需要额外的学习成本。

因此，虽然对于初学者来说，Angular 相对更难上手，但是我们实验室都具有它的相关基础，所以这方面不存在问题。

React 和 Vue 当时我也都不是很了解，所以我也只是先学了 Angular，就用了 Angular。

开发体验

由于我主要的技术栈是 Angular，所以我只能评价 Angular。

Angular 的开发体验是十分顺畅的，由于它原生支持 TypeScript，它的可维护程度是十分高的。

同时 Google 也为 Angular 集成了非常多方便的中间件，比如说跨域就集成了 http-proxy-middleware。

其实如果换做是 React，开发过程中引入 Rudex + TypeScript + Hooks 并进行重构，也不可谓轻吧。

Angular 的异步路由守卫是解决什么问题？

异步路由守卫，就是当用户访问某一个 URL 的时候，前端要判断 URL 是否应该被当前用户访问。

因为不同的用户有不同的权限，比如包括管理员，学生还有教师，学生肯定是不能访问课程管理界面。

所以就要判断用户是否有访问权限，如果不能访问就要提醒用户，并且跳转到首页。

Angular 在路由守卫这方面提供了一些钩子函数，我们的项目中主要使用了 CanActivate，就是当导航到某路由时，判断是否能被访问并做出对应操作。

我的具体做法是：

  1. 在 CanActivate 函数前加上 async，让它成为 async 函数，这样就可以在其中实现异步编程
  2. 把用户身份与当前模块信息发送给后端，让后端来判断当前模块是否可以被此用户访问
  3. 然后从后端拿到响应数据，并进行相关逻辑处理
  4. 最后在 CanActivate 中 return 一个布尔值，代表是否能够被访问

有没有了解过 Webpack 一些配置？

Webpack 是一个用于现代 JavaScript 应用程序的静态模块打包工具。

我们项目的打包本质上都是用到了 Webpack，不过基本上都是通过 CLI 直接生成打包文件的，具体的进阶配置我本人没有专门研究过。

Express 使用到了吗，怎么传给前端数据？

Express 我主要用来对 MySQL 做读写。我们学校有一些课后作业需要用到前后端交互，我主要是用这个来做后端。

Express 框架的核心是对 HTTP 模块的再包装，就等于在 HTTP 模块之上，加了一个中间层。

它的使用主要有三个步骤：

  1. 首先启动 Express，开始监听一个端口。
  2. 通过 use() 方法来注册中间件，实现简单的路由。
  3. 通过路由来展示一个 URL 请求，然后前端发起 HTTP 请求，返回 HTTP 响应报文。

AntV 是用来做什么的？

AntV 就是 Ant Design 的一部分，它主要用于数据可视化，就类似于 Echarts。

我们主要用它来将数据绘制成折线图，它里面提供了一个折线图的方法，我们就调用了它，然后绘制了折线图。

做页面有没有涉及到表单的操作？

有，主要是用了 Angular 的响应式表单。

它提供了一种模型驱动的方式来处理表单输入，其中的值会随时间而变化。

在使用响应式表单时，一个 FormControl 类的实例对应于一个表单控件。

在使用时，通过将控件的实例赋值给属性，之后可以通过监听它来跟踪表单控件的值和状态。

通过使用 FormControl 控件的 value 属性，可以获得当前表单控件的一份数据值拷贝；通过 setValue 方法则可以更新表单的控件值。

React 的 Hook 有使用过吗？

Hook 是 React 16.8 的新增特性。它可以在不编写 class 的情况下使用 state 以及其他的 React 特性。

在这之前，组件的标准写法是类，Hook 的出现使得组件的最佳写法变成了函数。

Hook 的设计目的，就是完全不使用类就能写出一个全功能的函数组件。

Hook 就是钩子，在函数组件中，如果需要外部功能和副作用，就用钩子把外部代码钩进来。

Hook 有很多，我基本上只用到了 useState()，它可以为函数组件引入状态。纯函数不能有状态，所以把状态放在钩子里面。

它返回一个数组，包含两个成员：

  • 第一个成员是一个变量，指向状态的当前值。
  • 第二个成员是一个函数，用来更新状态。

生命周期函数知道哪些？

生命周期函数有很多，基本上项目中常用的有这些：

Angular

  • ngOnInit，初始化组件时调用，最常用，用作异步加载数据
  • ngOnChanges，组件的数据绑定值更新时调用，频繁触发，需要注意对性能的影响
  • ngOnDestroy，组件销毁时调用

React

组件创建时：

  1. constructor，构造方法，用来做一些组件的初始化工作，如定义 this.state 的初始内容
  2. render，仅仅用于渲染的纯函数
  3. componentDidMount，组件挂载到 DOM 后调用，且只会被调用一次。最常用，用作异步加载数据

组件更新时：

  1. render，仅仅用于渲染的纯函数
  2. componentDidUpdate，在组件完成更新后立即调用，频繁触发，需要注意对性能的影响

组件销毁时：

  1. componentWillUnmount，组件销毁时调用

组件之间怎么通信？

React

父组件向子组件通信

React 是单向数据流，父组件数据一般通过 props 传递到子组件。

不能在子组件中修改 props 的值，如果父组件的某个 props 改变，React 会重新渲染所有子节点。

子组件向父组件通信

子组件向父组件通信也是通过 props。父组件通过 props 传递回调函数给子组件，子组件调用回调函数，传入参数给父组件实现通信。

兄弟组件通信

有两种方法：

  1. 子组件 A 传值给父组件，父组件再传给子组件 B。
  2. 使用 Redux。当 React 组件需要调用数据的时候，它就发起 Actions 并通知管理员（Store），Store 就去查找相关资料（Reducers），拿到返回的信息给组件。

Angular

@Input() 和 @Output() 是 Angular 提供的两个装饰器，它们为子组件提供了一种与父组件通信的方法。

父组件向子组件通信

@Input() 允许父组件更新子组件中的数据，主要有三个步骤：

  1. 在子组件中导入 Input
  2. 在子组件中用 @Input() 装饰需要被子组件读取的属性
  3. 在父组件的 HTML 中将这个属性与子组件绑定

子组件向父组件通信

@Output() 允许子组件向父组件发送数据，主要有四个步骤：

  1. 在子组件中导入 Output 和 EventEmitter，EventEmitter 是 Angular 中用来发出自定义事件的类。
  2. 在子组件中用 @Output() 装饰需要被父组件读取的属性
  3. 在子组件的 HTML 中绑定这个属性
  4. 在父组件的 HTML 中将输入的事件与父组件内部方法绑定，实现传入参数

兄弟组件通信

有两种方法：

  1. 子组件 A 传值给父组件，父组件再传给子组件 B。
  2. 通过 Service 订阅，多个组件都对同一个 Service 进行依赖注入，调用同一内部方法即可获取数据。

写 React 的时候有没有做状态管理，比如说 Redux？

为了弥补 React 中麻烦的兄弟组件通讯方式，Redux 建立起一个中央机制，方便各组件之间的通讯。

当 React 组件需要调用数据的时候，它就发起 Actions 并通知管理员（Store），Store 就去查找相关资料（Reducers），拿到返回的信息给组件。

我写 React 主要是单页应用，基本上只用到了父组件向子组件通信，不需要复杂的组件通讯，所以对 Redux 相对了解并不多。

知道虚拟 DOM 吗，它和真实的 DOM 有什么区别

JSX 是 JavaScript 的一个语法扩展，它最终会被转译成 React.createElement。它返回的就是虚拟 DOM（Vistual DOM）。

虚拟 DOM 出现的原因：操作 JavaScript 对象很快，但是因为浏览器 DOM 十分复杂，所以操作它非常慢，耗性能。

可以通过实现高效的 diff 算法，对比出操作前后有差异的 DOM 节点，然后只对更改了的 DOM 进行更新。这样可以大大减少没有必要的 DOM 操作，进而大幅提升性能。

比对是同级的，当发现这一级的 DOM 不同时，会直接更新该节点及其所有子节点。这种同级比对策略是出于性能上的取舍。因为 DOM 操作很少出现跨层级的情况，对比所有节点代价太高。

举例：

有四个 <div> 分别有字符串 1234，想将它改成 5678，如果用传统的 DOM 方式，就是把这四个 <div> 删掉，再新增四个 <div>。

但是虚拟 DOM 会使用 diff 算法，发现它们只有 innerHTML 不一样，所以直接更改了四次 innerHTML，十分简洁快速。

脏检查

Angular 的脏检查机制（Dirty checking，脏检测）是在数据变化中，确保已经收集到所有的数据变化，真正要进行界面的更新的相关机制。

虽然脏检查和虚拟 DOM 是在不同阶段的不同概念，但是数据变化和对数据变化做出界面响应，这两个阶段是它们内在最本质的特性。

Angular 接入了 ZoneJS，由它监听了 Angular 所有的异步事件。Angular 收到通知后，就使用脏检查执行变化检测。

脏检查其实就是存储所有变量的值，每当可能有变量发生变化需要检查时，就将所有变量的旧值跟新值进行比较，如果不相等就更新对应视图，调用 Angular 内置的视图渲染和更新渲染。
