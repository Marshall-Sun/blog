    说下从输入网址到加载完成的流程

    1. 浏览器接收网址并处理（进程与线程、URL 解析、强缓存判断）

      1. 用户输入网址（URL）后回车，浏览器主进程通过相关规则进行URL 解析，得到并完善相关信息。
      2. 浏览器主进程通过进程间通信（IPC）将请求转发至网络进程。
      3. 网络进程获取到 URL，进行强缓存判断。若加载强缓存成功则直接返回资源。
      4. 若加载强缓存失败，网络进程向服务器发送 HTTP 请求（网络请求）。

    2. 发送网络请求到服务器（OSI 模型、DNS、HTTP 请求报文、跨域问题、TCP 头部、三次握手）

      1. 应用层
        • 网络进程请求 DNS 解析，将域名解析成 IP 地址
        • 构建并发送 HTTP 请求报文（数据包）
        • 需要注意是否跨域
      2. 表示层
      3. 会话层，如果是 HTTPS 请求就建立 SSL/TLS 连接
      4. 传输层
        • 将数据包加上 TCP 头部
        • 三次握手建立 TCP 连接
      5. 网络层
        • IP 寻址
        • 将数据包加上 IP 头部，形成 IP 数据报
      6. 数据链路层，封装成帧
      7. 物理层，利用物理介质传输

    3. 服务器接收请求（协商缓存、HTTP 响应报文）

      1. 物理层，利用物理介质接收
      2. 数据链路层，将 IP 数据报从帧中取出
      3. 网络层，解析出 IP 头部，传递得到的数据包
      4. 传输层，解析出 TCP 头部，传递得到的数据包
      5. 会话层
      6. 表示层
      7. 应用层
        • 对 HTTP 请求报文进行解析，获取请求头和请求体
        • 判断重定向（301/302），判断协商缓存
        • 发送 HTTP 响应报文，步骤可参考上一节

    4. 客户端与服务器交互（TCP 和 UDP、HTTP、浏览器存储与鉴权）

      TCP 三次握手建立连接后，客户端与服务器会进行一系列 HTTP 交互，交互完成后四次挥手断开连接。

    5. 浏览器渲染（DOM 树、CSSOM 树、渲染树、重绘与回流、分层与合成、绘制列表、栅格化）

      1. 网络进程将获取到的数据包进行解析，提交给渲染进程。

      2. 渲染进程开始渲染阶段：

        1. 将 HTML 文件转换为 DOM 树。
        2. 将 CSS 文件转换为 CSSOM 树。
        3. 将 DOM 树与 CSSOM 树合并得到渲染树
        4. 根据渲染树来进行布局（即初次回流）。
        5. 对渲染树进行分层，并生成分层树。
        6. 为每个图层生成绘制列表，并将其提交到合成线程。
        7. 合成线程将图层分成图块，光栅线程进行栅格化，将图块转换成位图。
        8. 渲染进程发送绘制图块命令给浏览器进程。

      3. 浏览器进程生成页面，显示到屏幕上。

    详情见 从输入网址到加载完成。

    说下 TCP 和 UDP 的区别

    TCP UDP
    英文名 Transmission Control Protocol User Datagram Protocol
    中文名 传输控制协议 用户数据报协议
    定义 面向连接的、可靠的、基于字节流的传输层通信协议 简单的面向数据报的通信协议
    是否连接 面向连接 无连接
    是否可靠 可靠传输 不可靠传输
    传输方式 面向字节流 面向报文
    连接对象个数 只能是一对一通信 支持一对一，一对多，多对一和多对多交互通信
    首部开销 首部最小 20 字节，最大 60 字节 首部开销小，仅 8 字节
    适用场景 适用于要求可靠传输的应用，如文件传输 适用于实时应用，如视频会议、直播

    具体来看，TCP 和 UDP 相比有三大核心特性：

    • 面向连接，连接指的是客户端和服务器的连接，在双方互相通信之前，TCP 需要三次握手建立连接，而 UDP 没有相应建立连接的过程。

    • 可靠性，TCP 花了非常多的功夫保证连接的可靠，这个可靠性体现在两方面：有状态、可控制。相应的，UDP 就是无状态，不可控的。

      • 有状态：TCP 会精准记录哪些数据发送了，哪些数据被对方接收了，哪些没有被接收到，而且保证数据包按序到达，不允许半点差错。
      • 可控制：当意识到丢包了或者网络环境不佳，TCP 会根据具体情况调整行为，控制发送速度或者重发。

    • 面向字节流，UDP 的数据传输是基于数据报的，这是因为仅仅只是继承了 IP 层的特性，而 TCP 为了维护状态，将一个个 IP 包变成了字节流。

    说下 TCP 报文头部字段

    源端口、目标端口

    如何唯一标识一个连接？答案是通过 TCP 连接的四元组——源 IP、源端口、目标 IP 和目标端口。

    那 TCP 报文怎么没有源 IP 和目标 IP 呢？这是因为在 IP 层就已经处理了 IP。TCP 只需要记录两者的端口即可。

    序列号（Sequence number，seq）

    序列号用于对字节流进行编号，这样可以保证数据包的正确顺序。序列号是一个长度 32 位的无符号整数。

    例如序号为 301，表示第一个字节的编号为 301，如果携带的数据长度为 100 字节，那么 [301, 400] 为序号 301 的数据长度，下一个报文段的序号应为 401。

    ISN（Initial Sequence Number，初始序列号），在三次握手的过程当中，双方会用过 SYN 报文来交换彼此的 ISN。

    ISN 并不是一个固定的值，因为源 IP 和源端口号都是很容易伪造的，当攻击者猜测 ISN 之后，直接伪造一个 RST 就可以强制连接关闭的，这是非常危险的。而动态的 ISN 大大提高了猜测难度。

    确认号（Acknowledgment number，ack）

    用来告知对方，小于确认号的所有字节已经全部收到，这也是接下来期望接收的序列号。

    例如 B 正确收到 A 发送来的一个报文段，序号为 501，携带的数据长度为 200 字节，因此 B 期望下一个报文段的序号为 701，B 发送给 A 的确认报文段中确认号就为 701。

    数据偏移

    表示 TCP 所传输的数据部分应该从 TCP 包的哪个位开始计算，实际上指的是 TCP 报头的长度。

    保留位

    用于以后的扩用。

    标志位

    6 个标志位，可选值为 0 或 1。

    • URG（Urgent），仅当取 1 时紧急指针字段才有效。表示此报文段中有紧急数据，应优先传送。
    • ACK（Acknowledgement），仅当取 1 时确认号字段才有效，否则无效。在连接建立后所有传送的报文段都必须把 ACK 置为 1。
    • PSH（Push），取 1 时表示数据包收到后应立即交给上层应用，不能缓存。
    • RST（Reset），取 1 时表明 TCP 连接中出现错误，必须强制断开连接。
    • SYN（Synchronous），取 1 时表示这是一个连接请求或连接确认报文，在建立连接时用来同步序列号。
    • FIN（Finish），取 1 时表明此报文段的发送方的数据已发送完毕，准备断开连接。

    窗口

    指接收端的窗口，即接收窗口。用来告知发送端自己所能接收的数据量，从而达到一部分流量控制的目的。

    检验和

    防止传输过程中数据包有损坏，如果遇到校验和有差错的报文，TCP 会直接丢弃并等待重传。

    紧急指针

    长度为 16 位，配合 URG 标志位一起使用。它只能标识一个字节的位置。这个指针指向了紧急数据最后一个字节的下一个字节。

    说下三次握手

    TCP 连接需要保证双方同时具备两样能力：发送能力和接收能力。于是必须使用三次握手。

    假设 A 为客户端，B 为服务器。

    1. 最开始双方都处于 CLOSED 状态。
    2. B 开始监听某个端口，进入 LISTEN 状态，等待客户端连接请求。
    3. A 向 B 发送连接请求报文，其中 SYN=1, ACK=0, seq=x，之后 A 进入 SYN-SENT 状态。
    4. B 收到连接请求报文，返回连接确认报文，其中 SYN=1, ACK=1, seq=y, ack=x+1，之后 B 进入 SYN-RCVD 状态。
    5. A 收到 B 的连接确认报文后，还要向 B 发出确认报文，其中 ACK=1, seq=x+1, ack=y+1，之后 A 进入 ESTABLISHED 状态。
    6. B 收到 A 的确认报文后，也进入 ESTABLISHED 状态。

    注意：SYN 是需要消耗一个序列号的，下次发送对应的 ACK 序列号要加 1。

    凡是需要对端确认的，一定消耗 TCP 报文的序列号。

    SYN 需要对端确认而 ACK 不需要，因此 SYN 消耗一个序列号而 ACK 不需要。

    为什么不是两次握手？

    根本原因：无法确认客户端的接收能力，有可能会浪费连接资源。

    如果是两次，A 发了 SYN 报文想握手，但是这个包滞留在了当前的网络中迟迟没有到达，TCP 以为这是丢了包，于是重传，新的两次握手建立好了连接。

    看似没有问题，但是如果连接关闭后这个滞留的包到达了 B 呢？这时候由于是两次握手，B 只要接收报文然后发送相应数据包，就默认建立了连接，但是现在 A 已经断开了，造成连接资源的浪费。

    为什么不是四次握手？

    三次握手的目的是确认双方发送和接收的能力，这已经足够解决问题，过多握手用处不大。

    说下四次挥手

    1. 最开始双方都处于 ESTABLISHED 状态。

    2. A 打算断开，发送连接释放报文，其中 FIN=1, seq=u，之后 A 进入 FIN-WAIT-1 状态，注意此时 A 也属于 HALF-CLOSE 状态，即无法向 B 发送报文只能接收。

    3. B 收到连接释放报文，返回确认报文，其中 ACK=1, seq=v, ack=u+1，之后 B 进入 CLOSED-WAIT 状态。

    4. A 收到 B 的确认报文后，进入 FIN-WAIT-2 状态。

    5. 当 B 也打算断开时，发送连接释放报文，其中 FIN=1, ACK=1, seq=w, ack=u+1，之后 B 进入 LAST-ACK 状态。

    6. A 收到后发出确认报文，其中 ACK=1, seq=u+1, ack=w+1，随后进入 TIME-WAIT 状态，等待足够长的时间（2 个 MSL，Maximum Segment Lifetime，报文最大生存时间）后释放连接。

      注意：在这段时间内如果 A 没有收到 B 的重发请求，那么表示确认报文成功到达，挥手结束，否则 A 重发确认报文。

    7. B 收到 A 的确认报文后立即释放连接。

    等待 2 个 MSL 的意义

    如果 A 不等待直接关闭，当 B 还有很多给 A 发的数据包在路上时，若 A 的端口此时被新的应用占用，就会接收到无用数据包，造成数据包混乱。

    所以，最保险的做法是等 B 发的数据包都失效再启动新的应用，那为什么等待 2 个 MSL 而不是 1 个？

    • 第一个 MSL 确保 A 的确认报文最终能到达 B。
    • 第二个 MSL 确保 B 没有发送重发请求。

    为什么不是三次挥手？

    B 接收到 A 的连接释放报文后往往不会立即返回 B 的连接释放报文，必须等到 B 所有的报文都发送完毕了才能发送。因此 B 先发一个确认报文表示已经收到 A 的连接释放报文，过一段时间才发 B 的连接释放报文。

    如果是三次挥手，等于 B 将确认报文和连接释放报文合一，这个时候较长的延迟可能会导致 A 误以为确认报文没有到达，从而让 A 不断重发连接释放报文，造成连接资源的浪费。

    为什么不是五次挥手？

    四次挥手已经足够解决问题，过多挥手用处不大。

    说下 HTTP 和 HTTPS 的区别

    • HTTP 协议是超文本传输协议，直接运行在 TCP 之上，信息是明文传输。

      HTTPS 是超文本传输安全协议，运行在 SSL/TLS 之上，SSL/TLS 运行在 TCP 之上。

    • HTTP 的连接很简单，是无状态的。

      HTTPS 是由 HTTP + SSL/TLS 构建的可进行加密传输、身份认证的网络协议，比 HTTP 安全，但更耗费服务器资源。

    • HTTP 和 HTTPS 使用的是完全不同的连接方式，用的默认端口也不一样，前者是 80，后者是 443。

    • HTTPS 协议需要到 CA 申请证书，一般免费证书很少，需要交费。

    HTTPS 如何保证安全

    通常认为，如果通信过程具备了四个特性：机密性、完整性，身份认证和不可否认，就可以认为是安全的。

    HTTPS 则为 HTTP 增加了上述四个特性，故被认为是安全的。这是因为 HTTPS 在 HTTP 下层的传输协议 TCP/IP 上添加了一层安全的 SSL/TLS 协议，收发报文不再使用 Socket API，而是调用专门的安全接口。

    混合加密

    虽然非对称加密没有“密钥交换”的问题，但因为它是基于复杂的数学难题，运算速度很慢。

    所以把对称加密和非对称加密结合起来：

    1. 在通信刚开始的时候使用非对称加密，首先解决密钥交换的问题。
    2. 然后产生对称加密使用的“会话密钥”，再用公钥加密。因为会话密钥很短所以慢一点也无所谓。
    3. 对方拿到密文后用私钥解密，取出会话密钥。这样，双方就实现了对称密钥的安全交换，后续就不再使用非对称加密，全都使用对称加密。

    用非对称算法加密对称算法的密钥，形成会话密钥，再用这个密钥进行传输，这就是混合加密。

    摘要算法

    实现完整性的手段主要是摘要算法，可以理解成一种特殊的压缩算法，它能够把任意长度的数据“压缩”成固定长度、而且独一无二的“摘要”字符串，就好像是给这段数据生成了一个数字“指纹”。如 MD5，SHA-1 等，SSL/TLS 使用的是 SHA-2。

    摘要算法保证了“数字摘要”和原文是完全等价的。所以，我们只要在原文后附上它的摘要，就能够保证数据的完整性。

    数字签名

    加密算法结合摘要算法，我们的通信过程可以说是比较安全了。但还有漏洞，就是通信的两个端点。

    黑客可以伪装成网站来窃取信息，也可以伪装成用户，向网站发送支付、转账等消息，导致网站没有办法确认身份。

    现实生活中，解决身份认证的手段是签名和印章，只要在纸上写下签名或者盖个章，就能够证明这份文件确实是由本人而不是其他人发出的。

    与现实中的签名、印章对应的就是非对称加密里的“私钥”，使用私钥再加上摘要算法，就能够实现“数字签名”，同时实现“身份认证”和“不可否认”。

    数字签名的原理其实很简单，就是把公钥私钥的用法反过来，之前是公钥加密、私钥解密，现在是私钥加密、公钥解密。

    数字证书和 CA

    到现在，综合使用对称加密、非对称加密和摘要算法，已经实现了安全的四大特性。但还有一个“公钥的信任”问题，怎么来判断这个公钥是可信的？

    可以找一个公认的可信第三方 CA（Certificate Authority，证书认证机构）。它就像网络世界的公证中心，具有极高的可信度，由它来给公钥签名，用自身的信誉来保证公钥无法伪造，是可信的。

    CA 对公钥的签名认证也是有格式的，不是简单地把公钥绑定在持有者身份上就完事了，还要包含序列号、用途、颁发者、有效时间等等，把这些打成一个包再签名，完整地证明公钥关联的各种信息，形成数字证书（Certificate）。

    小结

    1. 加密算法的核心思想是使用密钥加密密文，守住了密钥，也就守住了密文。
    2. 对称加密只使用一个密钥，运算速度快，密钥必须保密，无法做到安全的密钥交换。
    3. 非对称加密使用两个密钥：公钥和私钥，公钥可以任意分发而私钥保密，解决了密钥交换问题但速度慢。
    4. 把对称加密和非对称加密结合起来就得到了“又好又快”的混合加密，也就是 TLS 里使用的加密方式，这样就实现了机密性。
    5. 摘要算法用来实现完整性，能够为数据生成独一无二的“指纹”，常用的算法是 SHA-2。
    6. 数字签名是私钥对摘要的加密，可以由公钥解密后验证，实现身份认证和不可否认。
    7. 公钥的分发需要使用数字证书，必须由 CA 来颁发，否则就是不可信的。

    说下 HTTP 缓存

    浏览器缓存是浏览器对之前请求过的文件进行缓存，以便下一次访问时重复使用，节省带宽、提高访问速度、降低服务器压力。可分为强缓存和协商缓存。

    浏览器加载资源，根据 HTTP header 判断是否命中强缓存；

    • 若命中强缓存：浏览器直接从本地缓存中读取资源，不会将 HTTP 请求发送至服务器，返回状态码 200。

    • 未命中：服务器接受到 HTTP 请求，并通过 header 验证这个资源是否命中协商缓存。（这包括检查 HTTP header 中的 If-None-Match 字段中的 Etag，资源版本标识符。看服务器内部是否有该版本的资源，如果有就命中协商缓存）

      • 命中协商缓存：返回状态码 304，不携带资源，告诉客户端可直接从缓存中加载。
      • 未命中：返回状态码 200，携带资源。

    强缓存与协商缓存区别：强缓存会优先于协商缓存。强缓存不发送请求到服务器，协商缓存会发送请求到服务器。

    说下 HTTP 1.0/1.1/2.0 的区别

    HTTP 1.1 & HTTP 1.0

    • 长连接，一个连接可以重复在多个请求/响应使用，可以大大减少等待时间，因为在发出第一个请求后，双方不需要重新三次握手四次挥手。

      在 HTTP 1.0 中，Connection: keep-alive 没有官方支持只能手动开启，关闭时每次请求都要创建新的连接，属于短连接。

      在 HTTP 1.1 后，默认支持并开启 Connection: keep-alive，客户端和服务器建立连接后，连接就会被保持不会断开，当客户端发送另外一个请求时，就可以复用已建立的连接。

    • Host头，指明了请求将要发送到的服务器主机名和端口号。这能够使不同域名配置在同一个 IP 地址的服务器上。

      HTTP 1.0 认为每台服务器都绑定一个唯一的 IP 地址，因此请求消息中的 URL 并没有传递主机名。但随着虚拟主机技术的发展，在一台物理服务器上可以存在多个虚拟主机并且共享一个 IP 地址。

      HTTP 1.1 的请求/响应消息在 header 中都必须包含 Host 字段，且请求消息中如果没有 Host 字段可能返回语法错误 400。

    • 缓存处理，HTTP 1.1 提供了更多可选择的缓存头来控制缓存策略，比如 E-tag，If-Unmodified-Since，If-Match，If-None-Match；HTTP 1.0 只能使用 header 中的 If-Modified-Since 和 Expires。

    • 带宽优化与断点续传，HTTP 1.1 在请求头新增了 Range，它允许只请求资源的某个部分，即返回码是 206，这样不仅能优化带宽使用，也实现了 HTTP 1.0 不能实现的断点续传。

    HTTP 2.0 & HTTP 1.x

    • 头部压缩，由于 HTTP 1.x 头部字段带有大量信息，而且每次都要重复发送。HTTP 2.0 使用 HPACK 算法进行压缩；同时在客户端和服务器端保存之前发送的键值对以减少传输次数。

    • 二进制帧，HTTP 1.x 的解析是基于文本格式，存在天然缺陷。HTTP 2.0 使用了二进制格式，提升了效率性和健壮性。

    • 强化安全，HTTP 2.0 一般都跑在 HTTPS 上。

    • 多路复用，即每一个请求都用作连接共享。这样在一个 TCP 连接中，客户端和服务器可以同时发送多个请求和响应，提高了网络利用率。

    说下 HTTP 的请求报文 / 响应报文格式

    HTTP 请求报文

    一个 HTTP 请求报文由四个部分组成：请求行、请求头、空行、请求数据。

    请求行

    请求行由请求方法、URL、HTTP 协议版本与回车换行符（CR/LF）组成，前三者之间用空格分开。

    举例：GET / HTTP/1.1

    请求头

    请求头由键值对组成，每行一对，用英文冒号 : 分隔。请求头通知服务器有关于客户端请求的信息，典型的请求头有：

    • Host：请求的主机名 - www.baidu.com

    • Access-Control-Allow-Origin：允许跨域访问的源，或者为 *

    • Cache-Control：判断强缓存

    • Expires，Last-Modified：保存强缓存过期时间

    • If-None-Match：判断协商缓存，保存 Etag

    • If-Modified-Since：判断协商缓存，保存协商缓存过期时间

    • Authorization：在 Angular 项目中保存 JWT，用作令牌鉴权

    • Cookie：存储于客户端扩展字段，向同一域名的服务端发送属于该域的 Cookie - PSTM=1490844191; BIDUPSID=2145FF54639208435F60E1E165379255;

    • User-Agent：用户浏览器信息 - Mozilla/5.0 (Macintosh; Intel Mac OS X 10_12_3) AppleWebKit/537.36

    空行

    最后一个请求头之后是一个空行，回车符（CR）和换行符（LF），通知服务器以下不再有请求头。

    请求体

    请求数据不在 GET 方法中使用，而是在 POST 方法中使用。与请求数据相关的最常使用的请求头是 Content-Type 和 Content-Length。

    HTTP 响应报文

    一个 HTTP 响应报文由四个部分组成：状态行、响应头、空行、响应数据。

    状态行

    状态行格式： HTTP 协议版本、状态码、状态码描述、回车/换行符

    举例：HTTP/1.1 304 Not Modified

    响应头

    响应头由键值对组成，每行一对，用英文冒号 : 分隔。响应头允许服务器传递不能放在状态行的附加信息，这些域主要描述服务器的信息和 Request-URI 进一步的信息，典型的响应头有：

    • Server：包含处理请求的原始服务器的软件信息
    • Date：服务器日期
    • Content-Type：返回的资源类型 (MIME)
    • Connection：连接方式
    • close：连接已经关闭
    • keep-alive：连接已保持，在等待本次连接的后续请求
    • Cache-Control：缓存控制
    • Expires：设置过期时间
    • Set-Cookie：设置 Cookie 信息

    空行

    最后一个响应头之后是一个空行，发送回车符和换行符，通知浏览器以下不再有响应头。

    响应体

    服务器返回给浏览器的响应信息，包含数据。

    说下 RESTful API

    RESTful API 是目前比较成熟的一套 API 设计理论，它基于表现层状态转移（Representational State Transfer，REST）原则。

    它的主要内容是：

    • 看 URL 就知道要什么
    • 看 HTTP 请求方法就知道干什么
    • 看 HTTP 状态码就知道结果如何

    URL

    统一资源定位符（Uniform Resource Locator，URL，俗称网页地址或简称网址），一般包括六部分：

    • 传送协议（protocol），比如 HTTP/HTTPS/FTP
    • 服务器（host），通常为域名，有时为 IP 地址
    • 端口号（port），以数字方式表示，若为默认值可省略
    • 路径（path），以 / 分割每一个目录名称
    • 查询（query），即查询参数
    • 片段（fragment），以 # 为起点，一般用来定位

    请求方法

    • GET：从服务器取出资源（一项或多项）。
    • POST：在服务器新建一个资源。
    • PUT：在服务器更新资源（客户端提供改变后的完整资源）。
    • PATCH：在服务器更新资源（客户端提供改变的属性）。
    • DELETE：从服务器删除资源。

    HTTP 状态码

    1xx：指示信息 – 表示请求已接收，继续处理

    2xx：成功 – 表示请求已被成功接收

    3xx：重定向 – 要完成请求必须进行更进一步的操作

    4xx：客户端错误 – 请求有语法错误或请求无法实现

    5xx：服务器错误 – 服务器未能实现合法的请求

    常见状态代码、状态描述的说明如下：

    200 OK：客户端请求成功

    204 No Content：没有新文档，浏览器应该继续显示原来的文档

    206 Partial Content：客户端请求部分数据，服务器成功完成

    301 Moved Permanently：永久重定向，所请求的页面已经转移至新的 URL

    302 Found：临时重定向，所请求的页面已经临时转移至新的 URL

    304 Not Modified：客户端存在缓存并发出了一个条件性的请求，服务器告诉客户端缓存可以继续使用。

    400 Bad Request：客户端请求有语法错误

    401 Unauthorized：请求未经授权

    403 Forbidden：请求被禁止

    404 Not Found：请求资源不存在

    405 Method Not Allowed：请求方法不被允许

    500 Internal Server Error：服务器发生不可预期的错误

    501 Not Implemented：服务器不支持此功能

    502 Bad Gateway：服务器代理错误

    503 Server Unavailable：服务器忙，暂时无法响应服务
